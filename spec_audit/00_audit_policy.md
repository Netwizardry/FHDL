# 00. 감사 정책 (Audit Policy)

## 1. 기본 원칙
*   **통제성:** 명세서는 설명문이 아니라 구현과 테스트를 강제하는 통제 문서여야 한다.
*   **무결성:** 문서 간 충돌이 없어야 하며, 하나의 진실원칙(Single Source of Truth)을 준수해야 한다.
*   **검증 가능성:** 모든 요구사항은 수치나 조건으로 환원되어 테스트 가능해야 한다.

## 2. 결함 등급 체계
*   **Critical (치명):** 구현 불가능, 설계 로직 충돌, 데이터 파손 위험. 즉시 수정 필요.
*   **Major (중대):** 주요 기능 누락, 인터페이스 모호성, 비기능 요구사항 미달. 출시 전 수정 필수.
*   **Minor (보통):** 용어 불일치, 설명 부족, 예외 케이스 누락. 품질 저하 요인.
*   **Trivial (경미):** 단순 오타, 서식 오류, 링크 끊김. 문서 정리 수준.

## 3. 판정 기준
*   **적합 (Pass):** 해당 항목의 요구사항이 정밀하게 정의됨.
*   **부분 적합 (Partial):** 존재하나 수치나 조건이 모호함.
*   **부적합 (Fail):** 항목 자체가 누락되었거나 내용이 상충됨.
