# 17. 테스트 시나리오 및 검증 기준 명세

FHDL 프로그램의 모든 모듈은 본 명세에 정의된 시나리오를 통과해야 하며, 각 테스트는 재현 가능한 결과를 바탕으로 합격 여부를 판정합니다.

## 1. 테스트 분류 체계

| 코드 | 분류 (Category) | 검증 대상 |
| :--- | :--- | :--- |
| **T-SYN** | Syntax | 토큰화, 키워드 인식, 블록 구조 등 기본 문법 |
| **T-SEM** | Semantic | 중복 ID, 참조 무결성, 필수 속성 누락 등 의미적 올바름 |
| **T-NET** | Network | 트리/병렬 구조 변환, 고립 노드 및 공급원 없는 경로 검출 |
| **T-CAL** | Calculation | 유량 합산, 유속 계산, 손실수두 및 총 요구양정의 수치 정확성 |
| **T-AUTO** | Auto-Sizing | `auto` 키워드에 따른 관경, 펌프, 탱크 사양의 최적 선정 |
| **T-ERR** | Error/Recovery | 오류 발생 후 수정 및 재계산 흐름의 안정성 |
| **T-NFR** | Non-Functional | 장애 허용성, Fail-Safe, 결과 설명 가능성 등 품질 속성 |
| **T-INT** | Integration | 전체 시스템(입력~결과 출력)의 엔드투엔드 워크플로우 |

## 2. 테스트 케이스 구조

모든 테스트 케이스는 다음 4가지 필수 요소를 포함합니다.
1.  **입력 시나리오:** 검증에 사용될 FHDL 코드 또는 사용자 조작 단계.
2.  **기대 동작:** 시스템의 내부 상태 전이 및 진단 코드 발생 기대치.
3.  **검증 포인트:** 결과 리포트나 그래프 모델에서 확인해야 할 핵심 수치/상태.
4.  **합격 기준:** 성공과 실패를 가르는 명확한 수리적/논리적 기준.

## 3. 상세 테스트 시나리오 (Verification Matrix)

### 3.1 [S-TST-001] T-CAL-001: 정상상태 수압 평형 검증
*   **Ref:** `R02` (2-Pass 수리해석)
*   **Input:** 단일 펌프, 직렬 배관 2구간, 말단 노즐 1개.
*   **Expect:** Pass 1(유량 합산)과 Pass 2(손실 계산)를 통한 최종 말단 압력 산출.
*   **Criteria:** 수렴 잔차 $< 10^{-4}$, 계산된 압력이 이론값 대비 $\pm 1\%$ 이내.

### 3.2 [S-TST-002] T-ERR-005: NPSHa 및 캐비테이션 진단 검증
*   **Ref:** `R03` (NPSHa/캐비테이션 진단)
*   **Input:** 펌프 흡입측 수조 수위 낮음, 흡입관 마찰 손실 높음 설정.
*   **Expect:** $NPSHa < NPSHr$ 조건 발생 시 `WRN003` (Cavitation Risk) 경고 발생.
*   **Criteria:** 진단 패널에 `WRN003` 코드와 정확한 계산 수치 표시 여부.

### 3.3 [S-TST-003] T-SYN-001: Inverse Sync (역방향 동기화) 검증
*   **Ref:** `R07` (Inverse Sync)
*   **Input:** GUI Canvas에서 배관 `p1`의 `length`를 10m에서 15m로 수정.
*   **Expect:** 원본 `.fhd` 파일의 `p1` 블록 내 `length` 값이 15m로 자동 패치됨.
*   **Criteria:** 텍스트 파일의 서식(주석 등)이 유지되면서 수치만 정확히 변경될 것.

### 3.4 [S-TST-004] T-OPS-002: 저널 기반 상태 복구 검증
*   **Ref:** `R10` (저널링 기반 장애 복구)
*   **Input:** 데이터 수정 중 강제 종료(Process Kill) 후 시스템 재기동.
*   **Expect:** 기동 시 `.journal` 파일을 감지하여 마지막 성공 상태로 자동 복구.
*   **Criteria:** 복구 후 모델 데이터가 사고 직전의 `Dirty` 상태와 일치할 것.

### 3.5 [S-TST-005] T-NFR-001: 원자적 저장 (Atomic Save) 검증
*   **Ref:** `R05` (데이터 무결성)
*   **Input:** 저장 중 디스크 꽉 참(Full) 또는 권한 오류 발생 시뮬레이션.
*   **Expect:** 원본 파일이 손상되지 않고 이전의 유효한 상태(`Saved`) 유지.
*   **Criteria:** 원본 `.fhproj` 파일의 체크섬이 변경되지 않았음을 확인.

### 3.6 [S-TST-006] T-CAL-002: Hazen-Williams 공식 적용 검증
*   **Ref:** `R06` (Hazen-Williams 마찰 모델)
*   **Input:** `friction_model = HW`, `c_factor = 140` 설정된 배관 네트워크.
*   **Expect:** [FOR-HW-001] 수식을 적용하여 마찰 손실 산출.
*   **Criteria:** 동일 조건의 DW 대비 오차 범위를 확인하고 HW 공식 결과값과 일치 여부.

### 3.7 [S-TST-007] T-REP-001: 데이터 출처 (Provenance) 검증
*   **Ref:** `R11` (Provenance)
*   **Input:** 리포트 생성 및 특정 결과값(관경 등) 확인.
*   **Expect:** 리포트 내 선정 근거 필드에 적용된 공식 ID와 입력 참조가 포함됨.
*   **Criteria:** `Simulation_Summary.json`에 `provenance_map` 데이터 존재 여부.

## 4. 판정 기준 및 허용 오차

*   **수치 비교:** 유량 ±0.1%, 유속/손실/양정 ±1.0%의 상대 오차 허용.
*   **상태 전이:** 치명적 오류 발생 시 상태가 `ValidationFailed` 또는 `CalculationFailed`로 정확히 전이되어야 함.
*   **진단 코드:** 발생한 오류의 코드가 명세된 코드(SYNxxx, SEMxxx 등)와 일치해야 함.
